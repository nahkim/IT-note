# 📝 오디오 녹음 파이프라인 (콜백 → 버퍼 → 백그라운드 flush)

## 한 줄 요약
> "녹음 중엔 IO 콜백이 PCM을 메모리 버퍼에 쌓고, 별도 백그라운드 스레드가 2초마다 버퍼를 비워 WAV에 씁니다."

이 구조의 핵심은 **실시간 오디오 콜백을 절대 블로킹하지 않기 위해**, 콜백은 데이터를 버퍼에 '쌓기만' 하고 **느린 디스크 쓰기는 별도 스레드로 떠넘기는** 생산자-소비자 패턴이다.

## 핵심 내용

### 1. 문장 분해 — 무슨 일이 벌어지나
- **IO 콜백** — 마이크 입력 API가 새 오디오 조각이 준비될 때마다 불러주는 함수([[콜백]]). 여기서 원시 오디오 샘플([[PCM]])을 받는다.
- **메모리 버퍼에 쌓고** — 콜백은 받은 PCM을 [[버퍼]]에 **복사만 하고 즉시 반환**한다(생산자).
- **별도 백그라운드 스레드가 2초마다** — 다른 스레드가 주기적으로([[프로세스와 스레드]]),
- **버퍼를 비워 WAV에 씁니다** — 쌓인 데이터를 꺼내([[flush]]) [[WAV]] 파일에 append(소비자).

### 2. 왜 이렇게 나누나 (가장 중요한 이유)
- 오디오 입력 콜백은 **고우선순위 실시간 스레드**에서 호출된다. 정해진 시간 안에 반드시 반환해야 하고, **디스크 I/O·락·메모리 할당 같은 '시간이 얼마 걸릴지 모르는' 작업을 하면 안 된다.**
- 콜백이 잠깐이라도 블로킹되면 **다음 오디오 조각을 놓쳐(dropout/글리치)** 녹음이 튄다.
- 그래서 콜백은 **"버퍼에 복사하고 즉시 리턴"**만 하고, **디스크 쓰기(블로킹 가능)는 백그라운드 스레드**로 분리한다.

### 3. 생산자-소비자 + 링 버퍼
- **생산자** = 오디오 콜백(빠르게 채움), **소비자** = 백그라운드 스레드(주기적으로 비움).
- 둘이 **공유 버퍼**를 쓰므로 안전한 넘김이 필요([[동시성 문제]]). 오디오에선 **락-프리 링 버퍼(SPSC)**가 정석 — 콜백이 락을 잡으면 그 자체로 블로킹 위험이라, 원자적 인덱스로 락 없이 주고받는다.

### 4. 왜 하필 "2초마다" (배칭 간격의 트레이드오프)
- **너무 자주** 쓰면 → 디스크 I/O 시스템콜 오버헤드↑.
- **너무 드물게** 쓰면 → 버퍼(메모리)↑, **크래시 시 잃는 미저장 데이터↑**, 파일 반영 지연↑.
- **2초**는 그 사이의 타협점(오버헤드 ↔ 메모리·손실 위험).

### 5. WAV로 쓰기
- [[WAV]]는 PCM을 담는 컨테이너. **헤더(fmt: 샘플레이트·비트뎁스·채널)를 먼저 쓰고, data 청크에 PCM을 계속 append**.
- 헤더의 크기 필드(RIFF/data 길이)는 녹음이 끝나 총 길이를 알 때 **마지막에 채워 넣는다**(또는 스트리밍용으로 미리 예약).

### 6. 주의점
- **버퍼 오버플로** — 소비자가 느리면 버퍼가 차 **샘플 유실**. 버퍼 크기·flush 주기를 여유 있게.
- **크래시 안전성** — 마지막 flush 이후 버퍼에 있던 데이터는 손실 → 주기를 짧게 하거나 종료 시 강제 flush.
- **종료 처리** — 스레드 정리 + 마지막 flush + WAV 헤더 크기 확정.

## 코드 / 예시
```
[마이크] ─▶ (실시간 스레드)                        (백그라운드 스레드)
            IO 콜백: PCM을 링 버퍼에 복사·즉시 반환    2초마다: 버퍼 비워 WAV에 append
                    │  (생산자, 논블로킹)                 ▲   (소비자, 디스크 I/O 허용)
                    ▼                                     │
              ┌──────────── 공유 링 버퍼 ────────────┐
              │  ....PCM....PCM....PCM....           │──▶ [WAV 파일]
              └──────────────────────────────────────┘
```

## 기억할 문장 / 핵심 포인트
> "**오디오 콜백에서 파일을 쓰지 마라.** 콜백은 버퍼에 복사만 하고 즉시 반환, 디스크 쓰기는 다른 스레드로." — 실시간 오디오의 제1원칙.
> flush 주기는 **I/O 오버헤드 ↔ 메모리·크래시 손실**의 트레이드오프다.
