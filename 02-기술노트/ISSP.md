# 📝 ISSP — 이슈별 보안 정책 (Issue-Specific Security Policy)

## 한 줄 요약
ISSP는 **특정 기술·이슈**(이메일·인터넷·개인장비·AI 도구 등)를 "어떻게 안전하게 써야 하는가"를 구체적으로 규정한 보안 정책. 보안 정책 3계층(**EISP → ISSP → SysSP**) 중 **중간 계층**이다.

## 핵심 내용

### 1. 보안 정책 3계층 (NIST SP 800-14)
- **EISP (Enterprise Information Security Policy)** — 최상위. 경영진(CEO/CISO)이 수립하는 **조직 전체 보안 프로그램·철학·책임·범위**. 전략 방향이 바뀔 때만 개정. → "왜, 어떤 방향으로."
- **ISSP (Issue-Specific Security Policy)** — 중간. **특정 기술/이슈별 상세 지침**. → "이 기술을 이렇게 써야 한다."
- **SysSP (System-Specific Security Policy)** — 최하위. **개별 시스템·장비의 설정·절차**. 2종 — *관리적 지침(managerial guidance)* + *기술 명세(technical specification)*(예: 방화벽 ACL, 접근제어 목록). → "이 장비를 이렇게 설정한다."

### 2. ISSP란 — 다루는 이슈 예
인터넷·웹 사용, 이메일, **개인 장비(BYOD)**, 원격근무·VPN, 소셜미디어, 복사·인쇄 장비, 해킹·무단 접근 금지, 그리고 요즘은 **생성형 AI 도구 사용 정책**까지. "누가, 무엇을, 어디까지 허용/금지하는가"를 명확히 한다.

### 3. ISSP의 구성 요소 (전형 목차)
1. **목적(Statement of Purpose)** — 정책의 범위·적용 대상.
2. **허용된 사용(Authorized Uses)** — 누가 어떻게 써도 되는지.
3. **금지된 사용(Prohibited Uses)** — 하면 안 되는 것.
4. **시스템 관리(Systems Management)** — 모니터링·관리 책임.
5. **위반과 제재(Violations of Policy)** — 신고 절차·처벌.
6. **검토·개정(Policy Review & Modification)** — 정기 갱신 절차.
7. **책임 한계(Limitations of Liability)** — 위반자 보호 불가 등 면책.

### 4. 작성 접근법 3가지
- **개별 정책 다수** — 이슈마다 별도 문서. 관리가 번거로움.
- **단일 종합 문서** — 하나에 몰아넣음. 통제 쉽지만 유연성↓.
- **모듈식(권장)** — 공통 틀 + 이슈별 모듈. 균형이 좋아 실무에서 선호.

## 코드 / 예시
```
[EISP]  조직 전체 보안 철학·방향 (경영진)          "왜"
   │
   ▼
[ISSP]  이슈별 상세 지침 (이메일/웹/BYOD/AI…)      "이 기술을 어떻게"
   │
   ▼
[SysSP] 시스템별 설정·절차 (방화벽 ACL, 접근제어)   "이 장비를 어떻게"
```

## 기억할 문장 / 핵심 포인트
> "EISP는 **방향**, ISSP는 **특정 기술의 사용 규칙**, SysSP는 **장비의 구체 설정**." — 계층이 내려갈수록 구체적·기술적이 된다.
> ISSP의 핵심 가치는 ① 사용자에게 **명확한 지침**을 주고 ② 위반 시 **조직의 법적 보호** 근거가 된다는 것.
