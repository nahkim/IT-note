# 📝 웹사이트 보안 (Web Security)

## 한 줄 요약
웹사이트 보안은 **하나의 방어책이 아니라 여러 겹의 방어(defense-in-depth)**로, 흔한 위협([[OWASP Top 10]])을 전송·입력·인증·운영 각 층에서 막는 것. 원칙은 **"모든 입력·요청을 불신하라(zero-trust)".**

## 핵심 내용

### 1. 위협 지형 — 무엇을 막나 (→ [[OWASP Top 10]])
접근 통제 실패 · 인젝션(SQL/명령) · **XSS** · **CSRF** · **SSRF** · 보안 설정 오류 · 취약 의존성(공급망) 등. 2026년엔 **에이전트형 AI 공격**(자동 정찰·취약점 탐색·페이로드 생성)이 더해져 공격 속도가 빨라졌다.

### 2. 전송 보안 (→ [[HTTP와 HTTPS]])
- **HTTPS 전면 적용 + TLS 1.3**, 안전한 cipher suite, 인증서 자동 갱신.
- **HSTS**로 HTTPS 강제(다운그레이드 공격 차단).

### 3. 보안 헤더 (→ [[HTTP 헤더]])
- **Content-Security-Policy(CSP)** — 허용된 출처만 스크립트/리소스 로드 → **XSS 완화**의 핵심.
- **X-Frame-Options / frame-ancestors** — 클릭재킹 방지.
- **X-Content-Type-Options: nosniff**, **Referrer-Policy**, **Permissions-Policy**.

### 4. 입력·출력 처리 (인젝션·XSS의 근본 방어)
- **입력 검증(validation)** — 모든 사용자 입력은 불신, 허용 목록 기반 검증.
- **출력 인코딩(encoding)** — 화면에 내보낼 때 컨텍스트에 맞게 이스케이프 → XSS 차단.
- **파라미터라이즈드 쿼리 / ORM** — 문자열 조립 대신 바인딩 → **SQL 인젝션** 차단. (→ [[관계형 데이터베이스와 SQL]] · [[ORM]])

### 5. 인증·세션 (→ [[인증과 인가]] · [[쿠키 세션 JWT]] · [[OAuth]])
- 쿠키에 **HttpOnly · Secure · SameSite** 속성. CSRF 토큰 또는 SameSite로 **CSRF** 방지.
- 강한 인증(**MFA**), 안전한 세션 만료·회전, 최소 권한([[RBAC]]).

### 6. 경계·운영
- **CORS** 정확히 설정 — 필요한 출처만 허용(와일드카드 남발 금지). (→ [[CORS]])
- **의존성·공급망** — 라이브러리 취약점 스캔·업데이트, 빌드 파이프라인 무결성. (→ [[CI-CD]])
- **WAF · 레이트리밋 · DDoS 방어**, 시크릿 관리(코드에 키 금지, → [[환경 변수]]), **로깅·모니터링**으로 탐지·대응.

## 코드 / 예시
```
[사용자 요청]
   │  ① HTTPS/TLS1.3 + HSTS (전송 암호화)
   ▼
[WAF · 레이트리밋]  ② 악성 트래픽 차단
   ▼
[앱]  ③ 입력 검증 · 출력 인코딩 · 파라미터라이즈드 쿼리
      ④ 인증(MFA)/세션(HttpOnly·Secure·SameSite)/접근통제(RBAC)
      ⑤ 보안 헤더(CSP·X-Frame-Options…) 응답에 부착
   ▼
[DB]  최소 권한 계정 · 암호화 저장
─ 전 구간: 로깅·모니터링 + 의존성 스캔(공급망) ─
```

## 기억할 문장 / 핵심 포인트
> "**입력은 전부 불신, 출력은 전부 인코딩.**" — 인젝션·XSS의 90%는 여기서 갈린다.
> "보안은 은탄환 하나가 아니라 **여러 겹(defense-in-depth)**. 한 층이 뚫려도 다음 층이 막는다."
> "[[OWASP Top 10]]은 최소 기준선일 뿐, 통과가 곧 안전은 아니다."
