# 📝 시스템 설계 기초 (System Design Fundamentals)

## 한 줄 요약
사용자가 늘어나도 **빠르고, 죽지 않고, 데이터가 깨지지 않게** 만드는 것이 시스템 설계의 목표다.

## 핵심 내용

### 1. 무엇을 고민하는가 (설계의 3대 축)
1. **확장성(Scalability)** — 트래픽이 10배 늘어도 버티는가? → [[수평 확장]]
2. **가용성(Availability)** — 일부가 고장 나도 서비스가 계속되는가? → [[로드 밸런서]]
3. **성능(Performance)** — 응답이 빠른가? → [[캐시]]

### 2. 트래픽이 늘면 거치는 단계
1. 서버 1대 → 한계가 옴
2. 서버를 **여러 대로 복제**하고 앞에 [[로드 밸런서]]를 둬서 요청을 분산
3. 자주 읽는 데이터는 [[캐시]]에 올려 DB 부하를 줄임
4. DB 한 대로 부족하면 데이터를 쪼개는 [[데이터베이스 샤딩]] 적용
5. 무거운 작업(이메일 발송 등)은 [[메시지 큐]]로 비동기 처리

### 3. 피할 수 없는 트레이드오프
- 데이터를 여러 서버에 나누는 순간 **[[CAP 정리]]**와 마주친다: 일관성·가용성·분단내성 중 둘만 강하게 가질 수 있다.
- "정답"이 아니라 **상황에 맞는 선택**이 설계다.

## 코드 / 예시
```
[사용자]
   │
   ▼
[로드 밸런서] ──▶ [웹서버 #1] ┐
              ──▶ [웹서버 #2] ├──▶ [캐시] ──▶ [DB (샤딩)]
              ──▶ [웹서버 #3] ┘                  
                                    무거운 작업 ──▶ [메시지 큐] ──▶ [워커]
```

## 기억할 문장 / 핵심 포인트
> "확장은 보통 더 좋은 서버 1대가 아니라, **평범한 서버 여러 대**로 푼다." ([[수평 확장]])
> "캐시는 만능이지만, **무효화(언제 지울지)**가 가장 어렵다."
