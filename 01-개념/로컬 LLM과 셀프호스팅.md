# 로컬 LLM vs 직접 모델 배포 (셀프호스팅)

## 한 줄 정의
둘 다 **"제3자 API(OpenAI 등) 대신 모델을 내가 굴린다"**는 점은 같다. 차이는 **어디서·누구를 위해** 굴리느냐 —
- **로컬 LLM** = 내 **개인 컴퓨터**에서(주로 1인·개발용) 실행.
- **직접 모델을 올린다(셀프호스팅)** = 내가 운영하는 **서버 인프라**에 올려(주로 다중 사용자·프로덕션) 서빙.

## 자세히

### 공통점 (둘 다 vs 매니지드 API)
- 모델 가중치를 **내가 갖고 실행** → 데이터가 외부 API로 나가지 않음(프라이버시), **사용량 과금 없음**, 오프라인 가능.
- 반대편은 OpenAI·Anthropic API처럼 **남이 호스팅해 주는** 방식(편하지만 데이터·비용·통제는 제공자에 의존).

### 로컬 LLM — 개인·데스크톱
- **도구**: **Ollama**(단일 바이너리, 1인·1머신, OpenAI 호환 API) · **LM Studio**(GUI, Hugging Face 모델 브라우저) · **llama.cpp**(C/C++, 어디서나 실행) · Apple **MLX**.
- **포맷**: 주로 **GGUF**(가중치·토크나이저를 한 파일에 묶은 포맷) + 양자화(Q4 등)로 작게.
- **특징**: 소비자 하드웨어(노트북·개인 GPU), **단일 사용자**, 빠른 프로토타이핑. → [[온디바이스]]와 인접.

### 직접 모델 올리기 — 서버·프로덕션
- **도구**: **vLLM**(고처리량·다중 사용자·PagedAttention+연속 배칭) · **SGLang**. (Hugging Face **TGI는 2025-12부터 유지보수 모드** → 신규는 vLLM/SGLang 권장)
- **인프라**: GPU 서버([[온프레미스]] 또는 자체 클라우드), [[nvidia-smi]]로 자원 감시.
- **특징**: **다중 동시 요청**, 프로덕션 처리량·안정성. 벤치마크상 vLLM이 Ollama 대비 동시 처리량 **~16~20배**, KV 캐시 낭비도 훨씬 적음(<4% vs 30~50%).

### 실무 경로 & 함정
- **프로토타입은 Ollama(로컬)로 빠르게 → 성숙하면 vLLM(셀프호스팅)으로 이관**이 흔한 경로.
- **포맷 우주가 다름**: llama.cpp/Ollama = GGUF, vLLM = safetensors(AWQ/GPTQ/FP8). 섞으면 성능 손해 → 처음부터 목적에 맞는 쪽 선택.

## 왜 중요한가
- "API를 쓸까 · 로컬로 돌릴까 · 서버에 올릴까"는 **비용·프라이버시·규모**의 핵심 결정.
- **개인 실험(로컬)과 프로덕션 서빙(셀프호스팅)을 혼동하면** 하드웨어·도구 선택이 어긋난다 — 1인용 Ollama로 다중 사용자 서비스를 돌리면 처리량이 무너진다.
