# 폴링 (Polling)

## 한 줄 정의
클라이언트가 서버(또는 자원)에 **주기적으로 반복 요청**해 "새 데이터·상태 변화 있어?"를 스스로 확인하는 방식. 서버가 먼저 알려주는 **'푸시(push)'의 반대편(pull)**.

## 자세히

### 기본 폴링 (Short Polling)
- 클라이언트가 일정 간격(예: 5초)마다 요청 → 서버는 **즉시 응답**(새 게 없으면 빈 응답).
- **장점** — 구현 단순, 상태 없음(stateless), 방화벽·프록시 친화적.
- **단점** — 대부분 **빈 응답으로 낭비**, 지연이 최대 '간격'만큼 발생, 서버 부하 = (클라이언트 수 × 요청 빈도).
- 예 — `setInterval`로 API 재호출, curl 상태코드 반복 확인([[curl]] 헬스체크).

### 롱 폴링 (Long Polling)
- 클라이언트가 요청하면 서버가 **데이터가 생길 때까지 응답을 보류**(또는 타임아웃)했다가 응답 → 클라이언트는 받자마자 **곧바로 재요청**.
- **장점** — 빈 응답↓, **준실시간**(생기는 즉시 도착). WebSocket 이전의 실시간 표준.
- **단점** — 서버가 많은 연결을 붙잡고 있어야 함, 구현이 더 복잡, 여전히 HTTP 요청-응답을 반복.

### 푸시/이벤트 기반 대안 (폴링을 대체)
- **SSE (Server-Sent Events)** — 서버→클라이언트 **단방향** 스트림(HTTP, `Content-Type: text/event-stream`). 뉴스·주가·알림에 적합, 브라우저 `EventSource`가 재연결을 자동 처리.
- **WebSocket** — **양방향 전이중** 지속 연결([[WebSocket]]). 채팅·게임처럼 클라이언트도 자주 보낼 때.
- **웹훅(Webhook)** — 서버 간 통신: 이벤트가 나면 상대 서버 URL로 직접 POST → 받는 쪽이 폴링할 필요가 없음.
- **실무 휴리스틱** — 저빈도·약간의 지연 허용 → **롱 폴링**, 단방향 잦은 갱신 → **SSE**, 양방향 저지연 → **WebSocket**. (폴링은 흔히 **폴백**으로 남겨둠)

### 다른 맥락의 '폴링' (웹 밖에서도 등장)
- **인터럽트 vs 폴링(OS·하드웨어)** — CPU가 장치 상태를 계속 확인(폴링/busy-wait) ↔ 장치가 신호를 보냄(인터럽트). 폴링은 단순하지만 CPU를 낭비.
- **I/O 멀티플렉싱** — `select`/`poll`/`epoll`로 여러 소켓의 준비 상태를 확인(이름은 poll이지만 효율적 이벤트 통지에 가까움).
- **풀(pull) 모델** — [[메시지 큐]]·CI 등에서 소비자가 주기적으로 가져오는 방식.

## 왜 중요한가
- **실시간성이 필요한데 푸시를 못 쓸 때의 기본 수단**. 지금도 폴백·단순 상태 확인·헬스체크에 널리 쓰인다.
- 핵심은 **간격(interval) 설정 = 지연 ↔ 부하 트레이드오프**. 너무 짧으면 서버가 폭주하고, 너무 길면 반응이 느리다.
- **"당기기(pull=폴링) vs 밀기(push)"** 는 시스템 설계의 근본 축이다.
