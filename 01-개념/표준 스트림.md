# 표준 스트림 (stdin · stdout · stderr)

## 한 줄 정의
프로그램이 시작할 때 기본으로 갖는 **3개의 입출력 통로**. **stdin(표준 입력)**은 데이터를 '읽는' 기본 통로(기본 소스는 키보드), **stdout**은 정상 결과를 '쓰는' 통로(화면), **stderr**는 에러를 쓰는 **별도** 통로. 각각 **파일 디스크립터 0·1·2**.

## 자세히

### stdin (표준 입력) — 이 노트의 주인공
- 프로그램이 입력을 **읽어오는 기본 스트림**(FD **0**). 기본 소스는 터미널(키보드)이지만, **리다이렉션·파이프로 파일이나 다른 프로그램의 출력**으로 바꿀 수 있다.
- 코드에서: Python `sys.stdin`·`input()`, C `scanf`/`read(0, ...)`, 셸 `read`.
- **EOF(입력의 끝)** — 터미널에선 **Ctrl+D**로 EOF를 보낸다. 파이프에선 **보내는 쪽 프로세스가 끝나면** EOF. 읽기는 보통 **블로킹**(입력이 올 때까지 대기 → [[동기와 비동기]]).

### 3형제와 파일 디스크립터
| 스트림 | FD | 기본 | 용도 |
|---|---|---|---|
| **stdin** | 0 | 키보드 | 입력 읽기 |
| **stdout** | 1 | 화면 | 정상 출력 |
| **stderr** | 2 | 화면 | 에러·로그(정상 출력과 **분리**) |
- stderr를 분리하는 이유: 정상 결과(stdout)를 **파이프로 넘기면서도** 에러는 화면에 **따로** 보이게 하려고.

### 리다이렉션 (통로 바꾸기)
- `프로그램 < in.txt` — stdin을 파일에서 읽기.
- `프로그램 > out.txt` — stdout을 파일로(덮어쓰기), `>>` 이어쓰기, `2> err.txt` — stderr만, `> f 2>&1`(또는 `&>`) — 둘 다.
- 예: `sort < names.txt > sorted.txt`.

### 파이프(`|`) — 프로세스를 잇는다
- 앞 프로그램의 **stdout → 뒤 프로그램의 stdin**으로 연결. 예: `ps aux | grep python`.
- 커널 버퍼(≈4KB)를 통해 흐르는 **생산자-소비자** 구조(→ [[오디오 녹음 파이프라인]]의 그 발상).

### 버퍼링 (자주 걸리는 함정)
- **stderr는 버퍼링 없음(즉시 출력).** **stdout은 터미널이면 줄 단위(line-buffered)**, **파이프/파일이면 블록 단위(fully-buffered, ~4KB)**.
- 그래서 파이프로 넘기면 출력이 **한참 안 보이다가 몰려 나오거나**, stdout/stderr **순서가 뒤섞일 수** 있다 → 필요하면 [[flush]]로 강제 출력한다. (→ [[버퍼]])

## 왜 중요한가
- **유닉스 철학의 핵심** — "한 가지를 잘하는 작은 프로그램들을 **파이프로 조합**한다". stdin/stdout이 그 **공용 인터페이스**.
- 결과는 stdout, 에러·로그는 stderr로 나누는 습관이 **자동화·디버깅**을 좌우한다(파이프라인엔 결과만 넘기고 에러는 따로 봄).
- 컨테이너·CI도 이 위에서 동작 — Docker `-i`는 컨테이너의 **stdin을 열어 두는** 옵션, 로그는 stdout/stderr로 수집([[컨테이너와 Docker]]).
