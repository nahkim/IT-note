# HTTP 헤더

## 한 줄 정의
HTTP 요청·응답에 본문과 별도로 붙는 **메타데이터(`이름: 값` 줄들)**. 핵심은 **클라이언트가 헤더를 자유롭게 붙여 보내고, 서버는 자기가 관심 있는 것만 골라 읽고 나머지는 무시한다**는 점.

## 자세히

- **헤더 = 요청에 딸려가는 부가 정보.** 본문(body)이 "무엇을 보내는가"라면, 헤더는 "이 요청을 어떻게 다뤄라"라는 메모다. `이름: 값` 형태로 여러 줄 붙는다.
- 실제로 와이어에 흐르는 요청은 그냥 텍스트 한 덩어리다:

  ```
  GET /jobs/123 HTTP/1.1
  Host: localhost:8000
  Authorization: Bearer sth_abc...   ← 서버가 읽음 (인증용)
  User-Agent: curl/8.0               ← 서버가 안 읽음 → 무시
  Accept: */*                        ← 안 읽음 → 무시
  ```

- **서버는 "도착한 글자"만 본다.** 누가(curl·앱·Swagger·Postman) 보냈는지는 전혀 모르고, 알 필요도 없다. 헤더 줄에 `Authorization:`이 있나 없나만 보는 식.
- **그래서 두 가지 성질이 나온다:**
  - 서버가 **안 읽는 헤더는 아무리 붙여 보내도 그냥 무시** → 요청이 깨지지 않는다. (함수/엔드포인트에 안 적힌 헤더를 보내도 에러 안 나는 이유)
  - 반대로 서버가 **꼭 읽으려는 헤더를 안 보내면** 거기서 막힌다. 예: `Authorization`을 빠뜨리면 인증 단계에서 `401`.
- **인증 헤더의 대표 형태**: `Authorization: Bearer <토큰>`. 서버가 이 줄에서 토큰을 꺼내 검증한다 → [[인증과 인가]] · [[OAuth]] · [[쿠키 세션 JWT]].

### FastAPI에서: 엔드포인트에 안 적혀도 인증되는 이유 — `Depends`가 먼저 실행

엔드포인트 함수에 `authorization`이 안 보여도 인증이 되는 건, 헤더 읽는 코드를 **의존성(dependency)으로 따로 빼서** `Depends`로 끌어 쓰기 때문이다 → [[FastAPI]].

```python
# 엔드포인트엔 authorization 이 없다
async def get_job(job_id, user_id = Depends(current_user)):
    ...

# 헤더는 이 "공용 부품" 안에서 읽는다
async def current_user(authorization: str | None = Header(default=None)):
    # "Bearer " 떼고 토큰 검증 → user_id 반환, 실패하면 401
    ...
```

**실행 순서가 핵심이다.** `Depends`로 걸린 함수는 엔드포인트보다 **먼저** 돈다:

```
요청 도착 (Authorization 헤더 포함)
   │
   ▼
① current_user 먼저 실행 ── 헤더 읽고 토큰 검사
   │
   ├─ 헤더 없음/틀림 → 401  (get_job 은 실행조차 안 됨)
   │
   └─ 통과 → user_id 확보
        │
        ▼
② 그제서야 get_job 실행 → 200 OK
```

즉 "서버가 헤더를 고른다"는 원리가 프레임워크 레벨에선 **`Depends`가 먼저 실행되며 그 안에서 필요한 헤더를 골라 읽는 것**으로 구현된다. 덕분에 인증 로직을 모든 엔드포인트에 복붙하지 않고 한 부품으로 공유한다.

### 같은 서버라도 "헤더를 붙이는 쪽"은 클라이언트마다 다르다

서버는 헤더만 보지만, **그 헤더를 요청에 실어 주는 건 클라이언트의 코드**다. 그래서 같은 백엔드라도 통로에 따라 결과가 갈릴 수 있다:

| 클라이언트 | `Authorization` 붙이는 법 | 결과 |
|---|---|---|
| 앱 (코드의 `fetch`) | 코드가 직접 헤더 추가 | ✅ |
| `curl` | `-H "Authorization: ..."` 로 직접 | ✅ |
| Swagger "Try it out" | 입력칸에 적어도 plain 헤더 파라미터를 안 실어줌\* | ❌ 401 |

\* Swagger의 실행기(swagger-client)는 `Authorization`을 **보안 스킴(Authorize 버튼) 전용**으로 예약해 둬서, 그냥 헤더 파라미터로 선언하면 와이어에 안 싣는다. FastAPI에서 `HTTPBearer` 같은 **보안 스킴**으로 선언하면 Authorize 버튼이 생기고 그때서야 제대로 보낸다. → Swagger의 "문서(OpenAPI 명세)"는 API에서 자동 생성되지만, "실행 버튼"은 별도의 JS HTTP 클라이언트라는 점을 구분해야 한다.

## 왜 중요한가
- "엔드포인트 코드엔 없는데 왜 인증이 되지/왜 401이지?" 같은 혼란이 **헤더는 따로 흐르고 서버가 골라 읽는다**는 한 가지 원리로 풀린다.
- curl은 되는데 Swagger는 안 되는 현상이 **버그가 아니라 클라이언트마다 헤더 조립 코드가 다른 것**임을 알 수 있다 → 디버깅 시 "어느 통로로 보냈나"부터 의심.
- 인증·[[CORS]]·캐시·콘텐츠 협상 등 웹의 거의 모든 동작이 헤더로 제어된다.
