# 온디바이스 모델 배포 (On-device Model Deployment)

## 한 줄 정의
학습된 모델을 **각 기기 안에서 돌아가도록 실제로 올리는 과정** — 기기용 포맷으로 변환·경량화하고, 앱에 동봉하거나 내려받아, 기기의 런타임으로 실행하게 만드는 것. [[온디바이스]]가 "왜/무엇"이라면, 이 노트는 **"모델을 기기에 어떻게 넣느냐"**를 다룬다.

## 자세히

### 큰 그림 — 서버 모델에서 기기 실행까지
**① 변환·최적화 → ② 런타임 선택 → ③ 패키징·배포 → ④ 업데이트 운영**

### ① 변환·최적화 (모델을 '기기용'으로)
- 서버용 모델(PyTorch/safetensors)을 기기용 포맷으로 변환: **Core ML**(`.mlpackage`) · **LiteRT**(구 TFLite, `.tflite`) · **ONNX**(`.onnx`) · **GGUF** · ExecuTorch(`.pte`).
- **경량화** — 양자화(INT8/INT4, GPTQ/AWQ), 프루닝, **증류(distillation)** 로 크기·메모리를 줄인다. (Apple이 Gemini를 증류해 온디바이스용 소형 모델로 만든 사례)
- 전형 워크플로: PyTorch 학습 → ONNX 내보내기 → 플랫폼별 Core ML/LiteRT로 변환. (ExecuTorch는 PyTorch에서 바로 `.pte`로 내보내 중간 변환을 생략)

### ② 런타임 (기기 위에서 모델을 돌리는 엔진)
- **Core ML**(Apple, Neural Engine 가속) · **LiteRT / LiteRT-LM**(Google, 모바일) · **ONNX Runtime**(크로스플랫폼) · **ExecuTorch**(Meta, PyTorch 네이티브 — 50KB 런타임·12+ 하드웨어 백엔드) · **llama.cpp / MLX / MLC LLM**(로컬 LLM 계열).

### ③ 패키징·배포 방식 (핵심 갈림길)
- **앱에 동봉(bundle/embed)** — 모델을 앱 바이너리에 포함. 설치 즉시 오프라인 사용 가능하지만 **앱 용량↑**, 업데이트는 앱 스토어를 통해.
- **온디맨드 다운로드** — 첫 실행/필요 시 서버에서 내려받아 기기에 저장. 앱은 가볍고 **모델만 따로 갱신** 가능.
- **OS 내장·공유 모델** — 플랫폼이 제공하는 모델을 API로 호출(앱이 모델을 직접 안 실어도 됨):
  - **Android AICore** — OS가 **Gemini Nano**를 백그라운드로 다운로드(~1GB)·관리하고 **여러 앱이 공유**. (시스템 레벨 온디맨드 배포)
  - **iOS Apple Foundation Models framework** — 모델이 **시스템에 내장**돼 있어 개발자가 따로 싣지 않고 API로 사용(`@Generable` 등). (OS 번들)

### ④ 업데이트·운영 고려사항
- 모델도 버전·[[릴리즈]] 관리 대상 → **OTA 업데이트·롤백**.
- **기기 파편화**(스펙 제각각), 저장공간·RAM·배터리·발열, **모델 보안**(가중치 추출·탈취 방지).

## 왜 중요한가
- "온디바이스로 하자"는 결정과 "모델을 기기에 **실제로 어떻게 넣느냐**"는 별개 문제다. **포맷·런타임·배포 방식** 선택이 앱 용량·업데이트 유연성·성능·프라이버시를 좌우한다.
- OS 공유 모델(AICore·Foundation Models) 덕분에 이제 **"앱마다 모델을 동봉"하지 않고도** 온디바이스 AI를 쓰기 쉬워지는 흐름.
