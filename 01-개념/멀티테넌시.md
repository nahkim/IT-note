# 멀티테넌시 (Multi-tenancy)

## 한 줄 정의
하나의 소프트웨어 인스턴스·인프라를 **여러 고객(테넌트, tenant)이 공유**하면서, 각 테넌트의 데이터·설정은 **논리적으로 격리**하는 아키텍처. SaaS의 핵심 설계다. ("테넌트" = 건물 세입자처럼 소프트웨어를 '임대'해 함께 쓰는 고객)

## 자세히

### 싱글테넌시 vs 멀티테넌시
- **싱글테넌트** — 고객마다 별도 인스턴스·인프라. 격리·커스터마이징 최강이지만 **고객이 늘수록 비용·운영 부담이 선형으로 증가**.
- **멀티테넌트** — 공유해서 **규모의 경제 + 업데이트 한 번에 전체 반영**. 대신 데이터 격리·성능 간섭·커스터마이징 제약이 숙제.

### 격리 모델 — Silo / Pool / Bridge (AWS SaaS Lens 분류)
- **Silo(사일로)** — 테넌트마다 **전용 인프라·DB·인스턴스**. 격리·규정준수 최강, 비용·운영 부담 최대(사실상 싱글테넌트를 여러 개 운영).
- **Pool(풀)** — 모든 테넌트가 자원을 **완전 공유**. 비용 효율·확장성 최고, 대신 격리와 **noisy neighbor**가 숙제.
- **Bridge(브리지)** — 둘의 혼합. 앱/일부 자원은 공유하되 민감한 부분(예: DB)은 분리. **현실의 대다수 SaaS**가 여기.

### 데이터 격리 패턴 (DB 관점) — 격리 강함 → 약함
1. **DB-per-tenant** (테넌트별 DB) — 물리적으로 완전 분리. 보안·규정준수·커스터마이징의 '골드 스탠다드'지만 자원 오버헤드 최대. → Silo.
2. **Schema-per-tenant** (테넌트별 스키마) — 한 DB 안에서 테넌트별 **네임스페이스(스키마)**로 분리. 중간 규모 SaaS의 균형점. → Bridge.
3. **공유 테이블 + `tenant_id`** — 같은 테이블에 **tenant_id 컬럼**으로 구분. 최저 비용·최고 확장성, 대신 격리를 코드/DB가 **강제**해야 함. → Pool.

### 격리를 '진짜로' 강제하기
- **Row-Level Security(RLS)** — DB가 `tenant_id` 기준으로 행을 자동 필터링. 앱 코드에만 의존하면 **`WHERE tenant_id = ?` 한 줄만 빠져도 다른 테넌트 데이터 유출** → RLS로 **DB 레벨에서 강제**하는 게 안전하다.
- **테넌트 식별** — 서브도메인(`acme.app.com`)·JWT 클레임([[쿠키 세션 JWT]]/[[토큰]])·요청 헤더 등으로 요청의 테넌트를 판별해 컨텍스트에 주입.
- **접근 제어** — [[인증과 인가]]·[[RBAC]]로 **테넌트 경계 + 테넌트 내 역할**을 함께 관리.

## 장점 / 도전과제
**장점**
- **비용 효율** — 자원 공유로 규모의 경제.
- **운영 단순** — 한 번 배포·패치하면 전체 테넌트에 반영.
- **확장성** — 새 테넌트 추가가 '레코드 추가' 수준으로 가벼움.

**도전과제**
- **데이터 격리·보안** — 유출 시 영향 범위(blast radius)가 전체 테넌트로 번짐.
- **Noisy neighbor(시끄러운 이웃)** — 한 테넌트의 폭주가 다른 테넌트 성능을 갉아먹음 → **쿼터·레이트리밋·성능 격리** 필요.
- **커스터마이징 제약** — 공유 스키마라 테넌트별 특수 요구를 반영하기 어렵다.

## 왜 중요한가
- SaaS 비즈니스의 **단가·마진을 좌우하는 근본 아키텍처 결정**. 격리 모델을 잘못 고르면 나중에 보안 사고나 비용 폭탄으로 되돌아온다.
- 본질은 **"얼마나 격리할까(안전·규정) ↔ 얼마나 공유할까(비용·운영)"**의 트레이드오프. 그래서 규제 산업은 Silo, 대규모 저가 서비스는 Pool로 기우는 경향.
- 클라우드 어디에나 등장 — [[쿠버네티스]] 네임스페이스 기반 격리, DB 샤딩, API 서비스 모두 멀티테넌시 문제.
