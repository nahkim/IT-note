# 계층형 데이터 모델링 (Hierarchical Data)

## 한 줄 정의
**부모-자식 관계로 이어지는 트리 구조 데이터**(조직도·카테고리·댓글 스레드·파일시스템 등)를 다루는 법. 특히 **평면(flat) 테이블뿐인 관계형 DB에 트리를 어떻게 담느냐**의 방식 선택이 핵심 — 방식마다 읽기/쓰기 비용이 다르다.

## 자세히

### 트리 기본 용어
- **루트(root)** — 부모 없는 최상위 노드(하나). **리프(leaf)** — 자식 없는 말단.
- **부모/자식**, **형제(sibling)**, **조상(ancestor)/자손(descendant)**, **하위 트리(subtree)**, **깊이(depth)/레벨**.
- 계층 데이터는 곧 [[자료구조|트리]]다. 문제는 이 트리를 **행·열(평면)** 로만 이뤄진 [[관계형 데이터베이스와 SQL|관계형 DB]]에 어떻게 표현하느냐.

### 네 가지 대표 방식 (Bill Karwin, *SQL Antipatterns*의 분류)

**① 인접 리스트 (Adjacency List)** — 각 행이 자기 **부모의 id**만 가리킨다. 가장 단순하고 공간 효율적.
```sql
CREATE TABLE node (id INT PK, parent_id INT, name TEXT);  -- parent_id → node.id
```
- 👍 삽입·이동이 **한 컬럼 수정**으로 끝난다. 직관적.
- 👎 "하위 트리 전체" 조회에 **재귀**가 필요 → [[재귀|재귀 CTE]](`WITH RECURSIVE`)로 해결.
```sql
WITH RECURSIVE tree AS (
  SELECT * FROM node WHERE id = 1              -- 시작 노드
  UNION ALL
  SELECT n.* FROM node n JOIN tree t ON n.parent_id = t.id
) SELECT * FROM tree;                          -- 1의 모든 자손
```

**② 경로 열거 (Path Enumeration / Materialized Path)** — 각 노드에 **루트부터의 경로를 문자열로** 저장(예: `/1/4/9/`).
- 👍 조상·자손 조회를 조인·재귀 없이 `LIKE '/1/4/%'` 한 번으로. 정렬도 쉬움.
- 👎 DB가 경로의 **참조 무결성을 강제 못 함**(문자열일 뿐). 노드를 옮기면 **하위 경로 전부 갱신**.

**③ 중첩 집합 (Nested Set)** — 트리를 전위 순회하며 각 노드에 **left/right 경계값**을 매긴다. 자손은 부모의 `left`~`right` 사이에 모두 포함.
```sql
-- 3의 모든 자손: 범위 비교 한 방, 재귀·조인 없음
SELECT c.* FROM node p, node c
WHERE p.id = 3 AND c.lft BETWEEN p.lft AND p.rgt;
```
- 👍 **하위 트리 읽기가 매우 빠름**(단순 범위 쿼리). 읽기 위주·거의 안 변하는 트리에 최적.
- 👎 삽입·삭제·이동 시 **경계값을 대량 재계산** → 쓰기가 비싸고 취약.

**④ 클로저 테이블 (Closure Table)** — 별도 테이블에 **모든 (조상, 자손, 깊이) 쌍**을 저장(자기 자신 포함).
```sql
CREATE TABLE tree_path (ancestor INT, descendant INT, depth INT);
```
- 👍 조상·자손 조회 **양방향 모두 인덱스로 빠름**, 삽입 비용은 중간, **엣지에 속성**(기간·관계 종류)도 붙일 수 있어 유연.
- 👎 경로 수만큼 행이 생겨 **저장 공간을 더** 쓴다.

### 방식 선택 가이드
| 접근 패턴 | 권장 |
|---|---|
| 쓰기 잦음 · 작은 부분만 조회(메뉴·게시판 스레드) | **인접 리스트**(+재귀 CTE) |
| 읽기 위주 · 안정적인 트리(카테고리) | **중첩 집합** |
| 읽기·쓰기 균형 · 유연성·엣지 속성 필요 | **클로저 테이블** |
- 실무 정석: **일단 인접 리스트로 시작**(단순·저공간)하고, 조회 성능이 문제 되면 다른 방식을 얹는다.

### 그 밖의 도구
- **재귀 CTE(`WITH RECURSIVE`)** — 인접 리스트를 표준 SQL로 순회하는 현대적 방법. PostgreSQL·SQL Server·Oracle·SQLite·MySQL 8.0+ 지원.
- **전용 타입** — SQL Server `hierarchyid`, PostgreSQL `ltree`(경로 열거의 확장판).
- **[[NoSQL|문서 DB]]** — 중첩 문서(JSON)로 계층을 그대로 담을 수 있어, 하위 트리를 한 문서로 통째 읽기 좋음(대신 부분 갱신·교차 조회는 불리).

## 왜 중요한가
- 조직도·카테고리 분류·댓글 스레드·파일시스템·권한 계층([[RBAC]])·부품표(BOM) 등 **계층 데이터는 어디에나** 있는데, 평면 테이블에 담는 순간 "방식 선택"이 성능을 좌우한다.
- 방식마다 **읽기 vs 쓰기 트레이드오프**가 뚜렷 → 접근 패턴(무엇을 자주 하는가)에 맞춰 고르는 게 설계 역량. [[자료구조]]에서 "무엇을 자주 하느냐로 구조를 고른다"는 원칙의 DB 버전.
