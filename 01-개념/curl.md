# curl

## 한 줄 정의
명령줄에서 HTTP 등 다양한 프로토콜로 요청을 보내는 클라이언트 도구. 질문한 명령은 **응답 본문은 버리고 HTTP 상태 코드만 출력**하는, 전형적인 "엔드포인트 살아있나?" **헬스체크** 패턴이다.

## 질문한 명령 한 줄 분해
```bash
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8082/stt
```

- **`curl`** — HTTP 요청을 보내는 클라이언트. 옵션이 없으면 기본은 **GET**.
- **`-s`** (`--silent`) — 진행률 막대·에러 메시지 같은 **curl 자체 출력을 끔**. 결과만 깔끔하게 보려고.
- **`-o /dev/null`** — 응답 **본문(body)**을 파일로 저장하는 옵션인데, 목적지가 `/dev/null`(휴지통)이라 **본문을 그냥 버린다**. STT 응답이 길거나 바이너리여도 화면을 더럽히지 않음.
- **`-w "%{http_code}\n"`** (`--write-out`) — 전송이 끝난 뒤 **포맷 문자열**을 출력. `%{http_code}`는 마지막 응답의 **HTTP 상태 코드**(예: 200·404·500), `\n`은 줄바꿈.
- **`http://localhost:8082/stt`** — 요청 대상 URL. **내 컴퓨터(localhost)의 8082 포트**에서 도는 서비스의 `/stt` 경로(= STT 서버 엔드포인트).

> 종합하면: **로컬 STT 서버(8082)의 `/stt`에 GET 요청을 보내고, 본문·잡음은 모두 버린 채 HTTP 상태 코드만 한 줄 찍는다.** 서버가 떠 있고 그 경로가 응답하는지 빠르게 확인하는 용도. (→ [[STT 음성 인식]])

### 출력값 해석
- `200` → 정상 응답.
- `000` → **연결 자체 실패**(서버가 안 떠 있거나 포트가 닫힘). curl이 응답을 못 받으면 코드가 `000`으로 찍힌다.
- `404` 경로 없음 · `405` 메서드 불일치(아래 주의) · `500` 서버 내부 오류 · `503` 일시 불가.

> ⚠️ `/stt`가 **POST 전용**이면 GET 요청에 `405`(Method Not Allowed)가 날 수 있다. 그땐 `-X POST`와 실제 페이로드(`-d`/`-F`)가 필요. 단순 '살아있나' 체크라면 상태 코드만으로 충분하다.

## 자세히 (조금 더)

### 왜 이렇게 쓰나 — 스크립트 헬스체크
성공 여부만 판단할 때 딱 맞는 형태. 예:
```bash
code=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8082/stt)
[ "$code" = "200" ] && echo "OK" || echo "DOWN ($code)"
```

### 유용한 `-w` 변수 (응답 속도 측정에 자주 묶음)
- `%{time_total}` — 전체 소요 시간(초)
- `%{time_namelookup}` — DNS 조회까지 / `%{time_connect}` — TCP 연결까지
- `%{time_starttransfer}` — 첫 바이트까지(TTFB) / `%{size_download}` — 받은 바이트 수

### 자주 같이 쓰는 옵션
- `-sS` — 조용히 하되 **에러는 보이게**(silent의 단짝)
- `-f` — 4xx·5xx면 **실패 종료코드** 반환(스크립트 분기용)
- `-L` — 리다이렉트 따라가기 · `-m 5` — 타임아웃(초)
- `-H` — 헤더 추가([[HTTP 헤더]]) · `-X` — 메서드 지정 · `-d`/`-F` — 본문 전송 · `-I` — 헤더만(HEAD)

## 왜 중요한가
- 서버·엔드포인트가 살아있는지, 무엇으로 응답하는지 **한 줄로 확인**하는 가장 흔한 디버깅·운영 습관.
- 본문을 버리고 상태 코드만 보므로 **빠르고 스크립트 친화적** — 헬스체크·배포 검증·모니터링·CI에 그대로 쓰인다.
