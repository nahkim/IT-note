# Ansible (앤서블)

## 한 줄 정의
서버 설정·소프트웨어 배포·작업 오케스트레이션을 **YAML로 코드처럼 자동화**하는 도구. 관리 대상에 에이전트를 깔지 않는 **에이전트리스(agentless)** 구성 관리 도구.

## 자세히

### 핵심 특징
- **에이전트리스** — 관리 대상(노드)에 Ansible을 설치하지 않는다. **SSH**(리눅스) 또는 WinRM(윈도우)로 접속해, 필요한 코드를 **임시로 보내 실행하고 지운다**. → 노드에 상주 프로세스가 없어 운영 부담이 적음.
- **푸시 방식(push)** — 제어 노드(control node)가 원하는 상태를 대상 노드들에 **밀어 넣는다**. (반대로 대상이 가져오는 pull 모델인 Puppet/Chef와 대비)
- **선언적 + 멱등성** — "이 상태가 되도록" 기술하면, 이미 그 상태면 아무것도 안 하고 `ok`, 바꿔야 하면 `changed`로 보고. 같은 플레이북을 **여러 번 돌려도 결과가 같다**([[멱등성]]).
- **YAML 기반** — 사람이 읽기 쉬운 설정 언어라 학습 곡선이 낮은 편. (Python으로 작성됨)

### 구성 요소
- **인벤토리(Inventory)** — "어떤 노드들을 관리하나" 목록. 호스트를 그룹(`[web]`, `[db]`)으로 묶는다.
- **플레이북(Playbook)** — 실행할 **작업(task) 목록을 담은 YAML 파일**. 어떤 호스트 그룹에 무엇을 할지 기술.
- **모듈(Module)** — 실제 일을 하는 코드 단위(예: `apt`, `copy`, `service`, `template`). 대상 노드로 보내져 실행됨. 대부분 **멱등하게** 설계됨.
- **태스크(Task)** — 플레이북 안에서 모듈 하나를 호출하는 한 단계.
- **롤(Role)** — 플레이북·변수·파일을 **재사용 가능한 묶음**으로 구조화한 것.
- **(참고) Ansible Galaxy** — 공유 롤/컬렉션 저장소.

### 동작 흐름 (한눈에)
1. 제어 노드에서 `ansible-playbook site.yml -i inventory` 실행.
2. 인벤토리에서 대상 호스트를 읽고 **SSH로 접속**.
3. 각 태스크의 모듈을 대상에 **임시 전송 → 실행 → 정리**.
4. 결과를 `ok / changed / failed`로 집계해 보고.

## 왜 중요한가
- 서버 수십~수백 대의 설정을 **손으로 SSH 들어가 반복**하는 대신, 코드 한 벌로 **일관되게 자동화**(IaC, Infrastructure as Code)한다.
- 에이전트리스라 도입 장벽이 낮음 — 대상에 SSH만 되면 시작 가능.
- 멱등성 덕분에 "지금 상태가 어떻든 원하는 상태로 수렴"시킬 수 있어, 배포·복구·일관성 유지에 강하다.
