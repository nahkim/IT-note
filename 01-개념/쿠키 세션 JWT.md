# 쿠키 · 세션 · JWT

## 한 줄 정의
[[HTTP와 HTTPS|HTTP]]는 매 요청을 기억 못 하는 **무상태**이므로, 로그인 같은 "상태"를 유지하려고 쓰는 세 가지 대표 방법.

## 자세히
- **쿠키(Cookie)** — 서버가 브라우저에 저장해 두는 작은 데이터 조각. 이후 요청마다 브라우저가 자동으로 다시 보낸다. (상태를 **실어 나르는 그릇**)
- **세션(Session)** — 상태를 **서버가** 기억하는 방식.
  - 로그인하면 서버가 세션을 만들고 **세션 ID**만 쿠키로 내려준다. 실제 사용자 정보는 서버(메모리·DB·[[캐시|Redis]])에 있다.
  - 서버가 상태를 갖는 **stateful** → 서버를 여러 대로 늘리면 세션 공유가 필요(→ [[수평 확장]] 시 고민거리).
- **JWT(JSON Web Token)** — 상태를 **토큰 자체에** 담는 방식.
  - 구조: `헤더.페이로드.서명` 세 부분을 **Base64url로 인코딩**해 점(`.`)으로 이은 문자열.
  - **서명(signed)일 뿐 암호화가 아니다** — 페이로드는 누구나 디코드해 읽을 수 있다. 서명은 **위변조 여부**만 검증한다. → 민감정보를 넣으면 안 됨.
  - 서버가 세션을 저장하지 않는 **무상태(stateless) 인증**이 가능해 [[수평 확장]]에 유리. 대신 발급 후 즉시 무효화(로그아웃)가 까다롭다.

## 왜 중요한가
- 로그인/세션 유지의 기본 3종. "세션(서버 기억) vs JWT(토큰에 담기)"는 거의 모든 백엔드 인증 설계의 갈림길이다.
- JWT를 "암호화된 안전한 것"으로 오해하면 보안 사고로 이어진다.

## 관련 개념
