# 클라이언트 앱 (Client App)

## 한 줄 정의
사용자의 기기에서 실행되며 서버에 요청을 보내는 쪽의 애플리케이션 — **클라이언트-서버 모델의 '클라이언트'**. 데스크톱·모바일·웹 앱 등이 모두 클라이언트다.

## 자세히

### 클라이언트-서버 모델
- **클라이언트**가 요청(request) → **서버**가 자원·처리를 제공(response). 보통 [[REST API]]·HTTP로 통신([[HTTP와 HTTPS]]).
- 클라이언트는 "사용자 접점(UI)"을, 서버는 "공유 데이터·무거운 처리"를 맡는 게 기본 분업.

### 씬 클라이언트 vs 씩 클라이언트
- **씬(thin) 클라이언트** — 처리를 서버에 의존하고 입출력 전달 위주(예: 브라우저에서 도는 웹앱). 기기 부담 적음.
- **씩(thick/fat) 클라이언트** — 기기에서 많은 처리·저장을 직접, **오프라인 가능**(예: 네이티브 데스크톱 앱).
- **하이브리드** — 중앙 데이터는 서버에 두되 로컬 처리도 함.

### 만드는 방식
- **네이티브** — Swift/Kotlin/C++ 등 플랫폼 언어. 성능·통합 최고, 플랫폼별 별도 개발.
- **웹** — 브라우저에서 실행(설치 없음). PWA·[[서비스 워커]]로 오프라인·"앱 같은 웹"도 가능.
- **웹뷰 기반 크로스플랫폼** — 웹 UI를 네이티브 셸로 감쌈. **[[Tauri]]**(웹 UI + Rust 백엔드를 OS 웹뷰로 묶음)가 대표 — Electron보다 **가볍고**(용량·메모리) 데스크톱+모바일을 한 코드베이스로. Electron은 Chromium 동봉으로 일관성↑·용량↑.

### 클라이언트 앱의 고려사항
- **오프라인·로컬 저장**, **온디바이스 처리**([[온디바이스]]) — 어디까지 기기에서 할지.
- **자동 업데이트** — [[릴리즈]] 채널로 새 버전 배포(Tauri 업데이터 등).
- **크로스플랫폼 빌드** — 대상 OS별 [[툴체인]] 필요.

## 왜 중요한가
- 사용자가 **실제로 만지는 접점**. **어디까지 클라이언트(기기)에서 하고 어디부터 서버에 맡길지**가 성능·오프라인·프라이버시를 좌우한다.
- [[Tauri]] 같은 도구 덕에 **웹 기술을 재사용하면서 가벼운 네이티브 앱**으로 배포하기 쉬워졌다.

## 관련 개념
- [[Tauri]] — 가벼운 크로스플랫폼 데스크톱·모바일 클라이언트 앱 프레임워크
- [[서비스 워커]] — 설치 없이 오프라인 되는 웹 클라이언트(PWA)
- [[REST API]] · [[HTTP와 HTTPS]] — 클라이언트↔서버 통신
- [[온디바이스]] — 클라이언트에서 직접 하는 처리
- [[릴리즈]] · [[툴체인]] — 클라이언트 앱의 배포·빌드
