# 소프트웨어 테스트 (단위 · 통합 · TDD)

## 한 줄 정의
코드가 의도대로 동작하는지 **자동으로 검증**하는 활동. 범위에 따라 단위·통합·E2E로 나뉜다.

## 자세히
- **테스트 피라미드** — 아래로 갈수록 많고 빠르게, 위로 갈수록 적게:
  - **단위 테스트(Unit)** — 함수·클래스 같은 **작은 단위 하나**를 검증. 밀리초 단위로 빠르고 많이 작성(피라미드의 바닥).
  - **통합 테스트(Integration)** — 여러 부분이 **함께** 잘 동작하는지(예: 서비스↔DB, API 간 통신).
  - **E2E 테스트(End-to-End)** — **사용자 관점에서 전체 흐름**(UI→API→DB)을 검증. 느리고 깨지기 쉬워 적게.
- **TDD(Test-Driven Development, 테스트 주도 개발)**: **테스트를 먼저** 작성하고, 그 테스트를 통과하도록 코드를 짜는 방식 → 테스트 가능성·설계 품질↑.
- [[CI-CD|CI/CD]]에서 코드를 푸시할 때마다 이 테스트들이 자동 실행돼 회귀(regression)를 막는다.

## 왜 중요한가
- "고쳤더니 다른 데가 깨졌다"를 막는 안전망. 자신 있게 리팩터링·배포하려면 테스트가 필수.
- 주니어가 좋은 코드를 짜는 습관의 핵심 — 테스트하기 쉬운 코드가 보통 좋은 설계다.
