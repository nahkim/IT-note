# CORS (Cross-Origin Resource Sharing)

## 한 줄 정의
브라우저가 기본으로 막는 **다른 출처(origin) 간 요청**을, 서버가 허용 헤더로 풀어주는 표준 규칙.

## 자세히
- **동일 출처 정책(Same-Origin Policy)**: 브라우저는 보안을 위해, 한 출처의 스크립트가 **다른 출처의 응답을 읽는 것**을 기본 차단한다. (출처 = 프로토콜+도메인+포트가 모두 같아야 동일)
  - 예: `https://app.com`의 JS가 `https://api.other.com`을 호출 → 기본 차단.
- **CORS**는 이를 안전하게 푸는 방법: 서버가 응답에 **`Access-Control-Allow-Origin`** 헤더로 "이 출처는 허용"이라고 알려주면, 브라우저가 응답을 통과시킨다.
- **프리플라이트(preflight)**: 부작용이 있을 수 있는 "복잡한 요청"(GET/단순 POST 외, 특정 헤더 등)은, 실제 요청 전에 브라우저가 **`OPTIONS` 요청**으로 "이 메서드·헤더 허용돼?"를 먼저 물어본다. 서버가 OK해야 본 요청을 보낸다.
- **중요한 오해**: CORS는 **브라우저가 강제**하는 규칙이다. 서버끼리(또는 curl·Postman) 직접 호출엔 적용되지 않는다. 또 "보안 강화"가 아니라 "동일 출처 정책의 **예외 허용**" 메커니즘이다.

## 왜 중요한가
- 프론트엔드↔백엔드를 다른 도메인/포트로 띄우면 가장 먼저 만나는 **"CORS 에러"**의 정체.
- 해결은 **서버 쪽에서** 허용 출처를 설정하는 것 — 프론트 코드로는 못 푼다는 점이 핵심.
