# 프롬프트 엔지니어링 vs 파인튜닝 (vs RAG) — 무엇을 언제

## 한 줄 요약
LLM을 내 목적에 맞추는 3가지 방법은 **무엇을 바꾸느냐**가 다르다: **프롬프트 엔지니어링 = 입력**, **RAG = 문맥(외부 지식 주입)**, **파인튜닝 = 모델 가중치**. 비용·효과가 커지는 순서이고, 실무는 **싼 것부터** 시작한다.

## 각 개념 (자세한 건 개념 노트)
- **[[프롬프트 엔지니어링]]** — 모델은 그대로 두고, **입력(프롬프트)을 설계·다듬어** 원하는 답을 유도. (지시·예시·형식·역할 부여 등)
- **[[RAG]]** — 질의 시점에 **외부 지식을 검색해 프롬프트에 끼워 넣어** 근거·최신성을 확보. ([[임베딩]]·[[벡터DB]] 기반)
- **[[파인튜닝]]** — 사전학습 모델을 **특정 데이터로 추가 학습**해 **가중치(행동·스타일)를 영구히** 바꿈. (LoRA 등 경량 방식 포함)

## 핵심 차이 — "무엇을 바꾸나"
- 프롬프트 엔지니어링: **입력만** 바꾼다(모델·지식 그대로).
- RAG: **모델에 주는 문맥(지식)**을 바꾼다(모델 가중치는 그대로).
- 파인튜닝: **모델 자체(가중치)**를 바꾼다.

## 비교표
| 기준 | 프롬프트 엔지니어링 | RAG | 파인튜닝 |
|---|---|---|---|
| 바꾸는 것 | 입력(프롬프트) | 문맥(검색된 지식) | 모델 가중치 |
| 지식 최신성 | 모델 학습 시점 고정 | **실시간·최신**(외부 소스) | 재학습 전까지 고정 |
| 비용·시간 | **거의 0 · 시간~일** | 중간(인프라·검색) | **높음 · 주~월**, 추론비↑ |
| 데이터 필요 | 거의 없음 | 문서 코퍼스 | **양질의 라벨 데이터셋** |
| 잘 맞는 경우 | 일반 작업·빠른 검증 | 사실 근거·사내 문서·챗봇 | 고정된 **스타일·형식·전문성·분류 정확도** |
| 약점 | 복잡·전문 작업 한계 | 검색 품질·지연에 의존 | 비용·유지보수·과적합 위험 |

## 언제 무엇을 (의사결정)
1. **프롬프트 엔지니어링부터** — 가장 싸고 빠름. 기능을 **검증([[PoC와 파일럿|PoC]])**하고 한계를 파악.
2. **한계가 오면 RAG** — "사실·최신 지식·사내 문서"가 부족할 때. 모델이 **모르는 것**을 문맥으로 채운다.
3. **그래도 안 되면 파인튜닝** — 프롬프트+RAG로도 **스타일·톤·추론 패턴·분류 정확도**가 기준에 못 미칠 때. 모델이 **해야 할 방식**을 새겨 넣는다.

> ⚠️ **'사다리'가 아니다** — 항상 순서대로만 가는 게 아니라, 데이터·지연·보안·품질 요구에 따라 **골라 쓰거나 조합**한다.

## 흔한 오해 / 조합
- "지식이 부족하다 → 파인튜닝"은 **자주 틀림**. 최신·사실 지식 문제는 대개 **RAG**가 답(파인튜닝은 지식 주입보다 **행동·형식 고정**에 강함).
- 실무 대세는 **하이브리드** — 좋은 프롬프트 + RAG(근거) + 필요 시 파인튜닝(스타일/전문성).
- 셋 다 안 되면 문제는 **모델 크기·데이터 품질**일 수 있음.
