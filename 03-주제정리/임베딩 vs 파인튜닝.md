# 임베딩 vs 파인튜닝 — 표현으로 풀까, 모델을 바꿀까

## 한 줄 요약
둘은 **경쟁 관계가 아니라 푸는 문제가 다르다**. **[[임베딩]]**은 의미를 벡터로 '표현'해 **찾고·비교·분류**하는 것(모델 안 바꿈). **[[파인튜닝]]**은 가중치를 바꿔 모델의 **생성 행동·스타일·전문성**을 바꾸는 것.

## 각각 무엇을 잘하나
- **임베딩** — 텍스트/이미지를 벡터로. **의미 검색·RAG 검색·분류·군집·추천·중복 제거**. 모델을 건드리지 않아 **싸고 빠르고 확장 쉬움**. ([[벡터DB]]에 저장)
- **파인튜닝** — 사전학습 모델을 데이터로 추가 학습해 **가중치 변경**. **일관된 스타일·톤·출력 형식·도메인 말투·특정 태스크(감성분석·NER 등)**에 강함. 대신 비용·시간·과적합 부담.

## 핵심 차이
- **바꾸는 것**: 임베딩=아무것도 안 바꿈(표현만 뽑음) / 파인튜닝=모델 가중치.
- **하는 일**: 임베딩=**찾기·비교(생성 아님)** / 파인튜닝=**생성 방식 바꾸기**.
- **지식 vs 행동**: 파인튜닝은 "새 정보 주입"보다 **"새 작업·행동 학습"**에 맞다. **사실·최신 지식은 임베딩+검색(RAG)**이 정석(파인튜닝 모델은 신뢰할 지식 저장소가 아님 → 환각).

## 비교표
| 기준 | 임베딩 | 파인튜닝 |
|---|---|---|
| 바꾸는 것 | 없음(표현 추출) | 모델 가중치 |
| 대표 용도 | 검색·분류·군집·추천·RAG | 스타일·형식·톤·태스크 특화 |
| 비용·시간 | **낮음** | 높음(추론비도↑) |
| 데이터 | 코퍼스(라벨 불필요 다수) | **양질 라벨 데이터셋** |
| 지식 최신화 | 소스만 갱신하면 됨 | 재학습 필요 |

## 언제 무엇을
- **찾기·유사도·분류·추천**이면 → **임베딩**(대부분 여기서 해결).
- **일관된 말투·형식·특정 태스크 정확도**가 필요하면 → **파인튜닝**.
- 헷갈리면: "**생성 결과를 바꿔야 하나(→파인튜닝)**, 아니면 **관련 걸 찾아오면 되나(→임베딩)**".

## 조합 (실무 대세)
- **RAG** = 임베딩(검색) + 프롬프트. 여기에 **출력 스타일이 중요하면 파인튜닝을 얹는다**.
- 임베딩으로 **군집 → 파인튜닝으로 분류 정제** 같은 파이프라인.
- 도메인 검색이 약하면 **임베딩 모델 자체를 파인튜닝**(embedding fine-tuning)하기도.

> 커스터마이징 '방법' 관점(프롬프트/RAG/파인튜닝)의 비교는 → [[프롬프트 엔지니어링 vs 파인튜닝]]. 이 노트는 **'표현·검색(임베딩)' vs '모델 변경(파인튜닝)'** 축에 집중.
