# REST vs WebSocket vs 폴링 (vs SSE) — 실시간 통신, 무엇을 언제

## 한 줄 요약
"서버의 변화를 클라이언트에 어떻게 전달하나"의 4가지: **REST(요청할 때만) · 폴링(반복해서 물어봄) · SSE(서버가 단방향 푸시) · WebSocket(양방향 상시 연결)**. 실시간성이 커질수록 연결을 오래 열고, 그만큼 확장이 어려워진다.

## 각 방식
- **[[REST API]] (HTTP 요청-응답)** — 클라가 물으면 서버가 답. **무상태**라 확장 쉽지만, 서버가 먼저 알릴 수 없어 실시간엔 부적합.
- **[[폴링]] (short/long)** — 클라가 **주기적으로 반복 요청**. 단순·범용, 대신 빈 응답 낭비·간격만큼 지연. 롱폴링은 응답을 보류해 준실시간.
- **SSE (Server-Sent Events)** — 서버 → 클라 **단방향 푸시**(HTTP 지속 연결). 알림·라이브 피드·대시보드에 최적(클라가 되보낼 필요 없을 때).
- **[[WebSocket]]** — **양방향 전이중** 상시 연결. 최저 지연. 채팅·게임·협업 편집. 대신 **상태ful**이라 클러스터 확장이 까다로움.

## 핵심 차이 — 방향과 연결
- **누가 먼저 말하나**: REST/폴링=클라가 물어야, SSE=서버가 밀어줌, WebSocket=양쪽 아무 때나.
- **연결**: REST/폴링=매 요청 새로, SSE/WebSocket=지속 연결.

## 비교표
| 기준 | REST | 폴링 | SSE | WebSocket |
|---|---|---|---|---|
| 방향 | 요청→응답 | 요청→응답(반복) | **서버→클라(단방향)** | **양방향** |
| 실시간성 | 낮음 | 간격만큼 지연 | 높음 | **가장 높음** |
| 연결 | 매번 | 매번 | 지속 | 지속 |
| 확장 | **쉬움**(무상태) | 쉬움(부하↑) | 비교적 쉬움 | 어려움(상태ful) |
| 적합 | 일반 CRUD API | 저빈도·폴백 | 알림·피드·진행률 | 채팅·게임·협업 |

## 언제 무엇을 (휴리스틱)
1. 단순 조회·명령이면 **REST**.
2. 가끔 갱신·간단하면 **폴링/롱폴링**. (단 **초당 1회 이상 폴링하면 WebSocket/SSE로 전환** 고려)
3. **서버가 밀어주기만** 하면 **SSE**(구현·확장 간단, 브라우저 자동 재연결).
4. **양방향·저지연**이 필요하면 **WebSocket**.
5. 서버↔서버 이벤트 통지는 **웹훅**(별개 축).

## 흔한 오해
- "실시간 = 무조건 WebSocket"은 과함. **단방향이면 SSE**가 더 싸고 단순.
- 폴링은 낡은 게 아니라 **저빈도·폴백**에 여전히 유효(헬스체크 등 → [[curl]]).
